電動車 (EV) 與電動車供電設備 (EVSE) 之間的通訊是實現安全、高效、智慧充電的基石。Vector Informatik 作為汽車電子開發工具和軟體領域的領導者，提供了全面的解決方案，支援從底層物理層到應用層的各類充電通訊協議。本報告將詳細分析Vector在EV/EVSE領域所支援的核心通訊協議，包括其技術定義、對應的硬體/工具，以及開發者在實作與測試過程中常遇到的技術門檻。

### **ISO 15118 (含前身 DIN 70121)**

#### 技術定義
ISO 15118 是電動車與充電基礎設施之間進行數位通訊的國際標準，旨在實現 Plug & Charge (即插即充)、V2G (車輛到電網) 和其他智慧充電服務。其核心是基於電力線通訊 (PLC) 或乙太網路的 IP 通訊協定棧，使用 XML 格式的訊息進行高層次握手與協商。

*   **ISO 15118-2:** 定義了基礎的 V2G 通訊協定，包括安全認證、充電參數協商、排程充電等，通常透過 HomePlug GreenPHY (PLC) 實現。
*   **DIN 70121:** 為 ISO 15118-2 的前身，德國國家標準，功能與邏輯類似，但訊息定義略有差異。
*   **V2G (Vehicle-to-Grid):** 允許電動車在特定條件下將電力回饋給電網，用於電網穩定、削峰填谷等應用。ISO 15118-2 包含 V2G 的基礎架構。
*   **Plug & Charge:** 簡化充電流程，用戶只需插入充電槍，車輛和充電樁自動完成認證、授權和計費，無需刷卡或App操作。這依賴於強大的PKI (Public Key Infrastructure) 安全機制。

#### 深度技術門檻分析

1.  **互通性 (Interoperability) 挑戰:**
    *   **標準解釋差異:** 儘管有標準，不同廠商對協議的解釋和實作可能存在微小差異，導致設備之間無法順利通訊。例如，SLAC (Signal Level Attenuation Characterization) 過程中的時序或參數調整。
    *   **測試複雜性:** 需要與多種不同廠商的 EV/EVSE 進行大量互通性測試，以確保穩定性。
2.  **安全性 (Security) 配置與管理:**
    *   **PKI/TLS 複雜性:** Plug & Charge 依賴於複雜的公開金鑰基礎設施 (PKI) 和傳輸層安全 (TLS)。證書管理、私鑰保護、證書鏈驗證等環節極其複雜且容易出錯。
    *   **攻擊面防護:** 需要防範中間人攻擊、重放攻擊等網路安全威脅，確保通訊加密和資料完整性。
3.  **時序敏感性與狀態機管理:**
    *   **協議流程複雜:** ISO 15118 的通訊流程包含多個階段和狀態切換，任何時序上的偏差都可能導致通訊失敗。
    *   **除錯困難:** PLC通訊的不透明性使得底層問題的診斷非常困難，需要專門的分析儀器。
4.  **HIL/SIL 測試環境建置:**
    *   **真實世界模擬:** 建立能夠準確模擬 EVSE 或 EV 行為的 HIL (Hardware-in-the-Loop) 或 SIL (Software-in-the-Loop) 測試環境，包括高壓功率級模擬，需要精密的硬體和軟體整合。
    *   **故障注入:** 模擬各種故障情境（如網路斷線、證書過期、非法訊息）以測試系統的魯棒性。

### **IEC 61851 / SAE J1772 (Control Pilot 訊號)**

#### 技術定義
IEC 61851 (全球標準) 和 SAE J1772 (北美標準) 定義了電動車充電系統的通用要求，特別是對於充電連接器、電纜以及 EV 和 EVSE 之間的低層次控制導頻 (Control Pilot, CP) 訊號。這些標準是所有交流 (AC) 和大部分直流 (DC) 充電協議的基礎。

*   **CP 訊號:** 透過充電連接器中的一個專用導線傳輸，EVSE 向 EV 發送脈衝寬度調變 (PWM) 訊號，指示最大可用電流。EV 則透過調整 CP 訊號上的電壓層級來指示其連接狀態 (例如，車輛已連接、準備充電、正在充電)。
*   **Proximity Pilot (PP) / 配對導頻:** 另一個導線，用於檢測充電線是否已插入車輛，並告知車輛充電線的額定電流容量。

#### 深度技術門檻分析

1.  **類比訊號的精準控制與測量:**
    *   **PWM 訊號品質:** CP 訊號的 PWM 方波必須精確產生和測量，包括頻率、佔空比和電壓幅值。任何偏差都可能導致誤判或通訊失敗。
    *   **雜訊抗干擾:** 充電環境通常存在較大電磁雜訊，確保 CP 訊號的穩定性和抗干擾能力是關鍵。
2.  **故障檢測與安全機制:**
    *   **絕緣監測:** 系統需要實時監測充電回路的絕緣狀態，防止漏電。
    *   **安全鏈驗證:** 確保所有安全相關的繼電器、傳感器和互鎖機制正常工作，特別是在高壓連接和斷開時。
3.  **HIL 測試中的真實環境模擬:**
    *   **負載模擬:** 在 HIL 測試中精確模擬車輛的電氣負載和充電行為，以驗證 CP 訊號在不同負載條件下的表現。
    *   **環境參數影響:** 模擬不同溫度、濕度等環境因素對 CP 訊號的潛在影響。

### **ISO 15118-20 (MCS, V2G)**

#### 技術定義
ISO 15118-20 是 ISO 15118 系列的最新主要版本，引入了多項重要增強功能，旨在支援未來的充電需求。

*   **MCS (Megawatt Charging System):** 專為重型電動車輛（如電動卡車、巴士）設計，支援高達數兆瓦的充電功率。它在電力傳輸層面帶來了全新的挑戰。
*   **更先進的 V2G 服務:** 擴展了 V2G 功能，包括更精細的能源管理、電網服務 (如頻率調節) 和多重電源協調。
*   **增強的協議效率與靈活性:** 引入了新的通訊機制和訊息格式，以優化通訊效率和適應性。
*   **支持新的連接器類型:** 除了現有的 CCS 連接器，ISO 15118-20 也考慮了 MCS 連接器。

1.  **兆瓦級功率傳輸與安全:**

    *   **熱管理:** 兆瓦級功率會產生大量熱量，需要先進的液體冷卻系統，測試時需評估其效率和可靠性。
2.  **複雜的 V2G 能源管理與電網互動:**
    *   **多方協調:** V2G 功能涉及 EV、EVSE、電網營運商、聚合商等多方實體的複雜互動，需要精確的排程和協調。
    *   **通訊負載:** 更多的 V2G 服務意味著更高的通訊頻率和資料量，對協議的效率和延遲要求更高。
3.  **新的協議定義與測試工具開發:**
    *   **標準成熟度:** 作為相對較新的標準，ISO 15118-20 的實作和測試工具仍在不斷發展和完善中，開發者可能面臨規範解釋不足或工具支援不完善的問題。
    *   **互通性再升級:** 更複雜的協議意味著更高的互通性風險，需要更為詳盡的測試場景。

### **GB/T 27930 (中國直流充電標準)**

#### 技術定義
GB/T 27930 是中國電動車直流充電的國家標準。它定義了電動車與非車載充電機之間的通訊協定，旨在確保充電過程的安全性和互通性。與 CCS 協議基於 PLC/Ethernet 不同，GB/T 的直流充電通訊主要基於 CAN (Controller Area Network) 總線。

*   **CAN 通訊:** 使用 CAN 總線傳輸控制訊息和充電參數，包括握手、充電請求、狀態回報、故障警報等。
*   **CP 訊號:** 同樣使用 CP 訊號進行低層次的連接檢測和絕緣監測，但其細節與 IEC 61851 有所不同。
*   **物理連接器:** 採用中國特有的直流充電連接器類型。

#### 深度技術門檻分析

1.  **CAN 總線的複雜性與診斷:**
    *   **訊息定義:** GB/T 27930 對於 CAN 訊息的 ID、資料內容和時序有嚴格定義，需要仔細解析 DBC (Database CAN) 文件。
    *   **總線負載與衝突:** 在 CAN 總線上傳輸大量訊息時，可能會出現總線負載過高、訊息衝突或延遲問題，影響充電效率和穩定性。
    *   **總線故障診斷:** CAN 總線的物理層問題（如線纜損壞、終端電阻不正確）或軟體問題（如錯誤的位元率）可能難以診斷。
2.  **與國際標準的差異和互通性:**
    *   **協議轉換:** 如果產品需要同時支援 GB/T 和 CCS 等國際標準，需要在設計和實作中考慮協議轉換的複雜性。
    *   **物理連接器差異:** 不同的連接器需要不同的硬體介面。
3.  **標準更新與解釋:**
    *   **規範演進:** 中國標準也在不斷更新和完善，需要及時跟進最新版本。
    *   **本地化支援:** 在實作和測試過程中，可能需要與當地供應商或測試機構合作，以理解標準的特定解釋和要求。

### **OCPP (Open Charge Point Protocol)**

#### 技術定義
OCPP 是一個應用層協議，主要用於充電站 (Charge Point) 與中央管理系統 (Central System / CSMS) 之間的通訊。它不直接涉及 EV 和 EVSE 之間的通訊，而是管理充電站的運營，如啟動/停止充電、韌體更新、遠端診斷、電價管理、用戶認證等。OCPP 允許不同廠商的充電站與不同的 CSMS 之間進行通訊，實現了互通性。

*   **OCPP 1.6 J (JSON):** 目前最廣泛採用的版本，基於 JSON 和 WebSocket 實現。
*   **OCPP 2.0.1 (JSON):** 最新版本，引入了對 Plug & Charge (透過 ISO 15118)、智慧充電、設備管理等更全面的支援，功能更強大。
*   **WebSocket/SOAP:** OCPP 協議的底層通訊機制，利用 TCP/IP 進行傳輸。

#### 深度技術門檻分析

1.  **後端系統整合與網路穩定性:**
    *   **複雜的系統架構:** OCPP 涉及充電站、CSMS、支付網關、能源管理系統等多個獨立系統的整合，任一環節出錯都可能導致服務中斷。
    *   **網路延遲與可靠性:** CSMS 通常是雲端服務，網路延遲、不穩定或斷線都會影響充電站的響應能力和功能。
2.  **安全通訊與鑑權:**
    *   **TLS 加密:** 確保 OCPP 通訊的安全性，實作和管理 TLS (Transport Layer Security) 證書是必須的。
    *   **用戶認證與授權:** OCPP 需處理 RFID 卡、App 等多種用戶認證方式，並與後台資料庫進行授權驗證。
3.  **協議版本差異與擴展性:**
    *   **版本演進:** OCPP 協議不斷演進，新舊版本之間存在功能差異，需要確保向下相容性或支援多版本。
    *   **廠商特定擴展:** 許多廠商會對 OCPP 進行非標準的擴展，這可能導致互通性問題。

### **Automotive Ethernet (10BASE-T1S, 100BASE-T1, 1000BASE-T1)**

#### 技術定義
車用乙太網路是一種專為車輛環境設計的乙太網路技術。它以單對非遮蔽雙絞線 (Single Pair Ethernet, SPE) 取代傳統的多對線纜，以實現輕量化、低成本、高頻寬和抗電磁干擾的能力。雖然它不直接是充電通訊協議，但在現代電動車內部，充電控制單元 (CCU) 會透過車用乙太網路與 BMS (電池管理系統)、VCU (車輛控制單元) 等進行高速通訊，對於充電流程的精確控制和數據交換至關重要。未來也可能被用於直接的 EV-EVSE 通訊。

*   **10BASE-T1S:** 適用於低速應用，例如感測器和執行器，能連接多個設備在同一條總線上 (類似 LIN 總線)。
*   **100BASE-T1:** 適用於中高速應用，如攝影機、雷達或 ECU 間通訊。
*   **1000BASE-T1:** 適用於更高頻寬需求，例如車載娛樂系統、ADAS 感測器融合。
*   **SOME/IP:** Service-Oriented Middleware over IP，在車用乙太網上實現服務導向的通訊。
*   **TSN (Time-Sensitive Networking):** 乙太網路標準的擴展，提供時間同步和確定性通訊，適用於實時控制應用。

#### 硬體與軟體對應

| OSI 階層       | Vector 硬體                                                               | 通訊介面               | Vector 軟體                                                         |
| :------------- | :------------------------------------------------------------------------ | :--------------------- | :------------------------------------------------------------------ |
| **應用層**     | VN系列 (如 VN56xx 系列, VN7200), VT系統 (帶有乙太網路模組)                | 10BASE-T1S/100BASE-T1/1000BASE-T1 | CANoe (乙太網路選項, SOME/IP Explorer), vTESTstudio, CANalyzer (乙太網路選項) |

#### 深度技術門檻分析

1.  **物理層測試與 EMI/EMC 挑戰:**
    *   **SPE 訊號完整性:** 單對乙太網路對線纜品質、連接器和終端匹配要求極高，確保訊號完整性是關鍵。
    *   **電磁相容性 (EMC):** 車內複雜的電磁環境對車用乙太網路的 EMI/EMC 性能構成挑戰，需要嚴格的測試和設計。
2.  **複雜的網路配置與管理:**
    *   **交換機配置:** 車用乙太網路通常涉及複雜的交換機網路，交換機的配置（VLAN、QoS、TSN 等）對網路性能影響巨大。
    *   **IP 地址管理與服務發現:** 在大規模車用乙太網路中，IP 地址分配、服務發現 (如 SOME/IP SD) 和網路拓撲管理變得複雜。
3.  **時間同步與實時性 (TSN):**
    *   **PTP 實作:** 為了實現確定性通訊，需要精確的時間同步 (如 PTP, Precision Time Protocol)。PTP 的實作和驗證是挑戰。
    *   **延遲與抖動:** 確保關鍵控制訊號的延遲和抖動在可接受範圍內，滿足實時性要求。

### **CAN / LIN (輔助車內通訊)**

#### 技術定義
CAN (Controller Area Network) 和 LIN (Local Interconnect Network) 是汽車中最常見的序列通訊總線。儘管它們不直接用於 EV 與 EVSE 之間的通訊，但它們在電動車內部扮演著至關重要的角色，用於傳輸充電控制單元 (CCU)、電池管理系統 (BMS)、電驅系統和車身電子裝置之間的數據。例如，BMS 可能透過 CAN 向 CCU 傳輸電池狀態、溫度和充電請求，CCU 則根據這些資訊與 EVSE 進行協商。

*   **CAN:** 廣泛用於動力總成、底盤和車身電子的多主從通訊，具有高可靠性、錯誤檢測和優先級仲裁。
*   **LIN:** 適用於低成本、低速的單主多從通訊，常用於車身電子中的簡單感測器和執行器。

#### 硬體與軟體對應

| OSI 階層       | Vector 硬體                                                               | 通訊介面       | Vector 軟體                                                         |
| :------------- | :------------------------------------------------------------------------ | :------------- | :------------------------------------------------------------------ |
| **應用層**     | N/A (由高層診斷協議如 UDS 或應用軟體處理)                               | N/A            | CANoe (應用層協議解釋), vTESTstudio                                 |
| **表達層**     | N/A (軟體處理訊息解析，例如 DBC 文件)                                   | N/A            | CANoe (DBC 解析), CANalyzer                                         |
| **資料連結層** | VN系列 (所有 VN 接口都支援 CAN/LIN), VT系統 (CAN/LIN 模組), GL系列 (CAN/LIN 數據採集) | CAN, LIN       | CANoe (訊息過濾、傳輸), CANalyzer (總線監測), Indigo (診斷)       |
| **實體層**     | VN系列 (CAN/LIN PHY), VT系統 (CAN/LIN PHY)                                | CAN (ISO 11898-2), LIN | CANoe (監測總線電壓), CANalyzer                                     |

#### 深度技術門檻分析

1.  **DBC / LDF 文件管理與解析:**
    *   **訊息定義:** CAN/LIN 訊息的詳細定義（ID、訊號位置、比例、偏移）儲存在 DBC (Database CAN) 或 LDF (LIN Description File) 文件中。這些文件的管理、版本控制和解析對於正確解讀數據至關重要。
    *   **複雜的訊號依賴:** 在複雜的電動車系統中，充電相關的 CAN/LIN 訊號可能涉及多個 ECU，且存在複雜的依賴關係，除錯困難。
2.  **總線負載與實時性:**
    *   **總線利用率:** 在高負載情況下，CAN 總線的利用率可能達到極限，導致訊息延遲或丟失。需要仔細設計訊息發送策略。
    *   **實時通訊:** 雖然 CAN 具有仲裁機制，但在特定的實時控制應用中，仍需確保關鍵訊息的確定性延遲。
3.  **診斷與故障注入:**
    *   **UDS 診斷:** 在 CAN 總線上實施統一診斷服務 (UDS, ISO 14229) 用於讀取故障碼、修改參數等。
    *   **故障模擬:** 模擬 CAN/LIN 總線上的各種故障（如訊息丟失、錯誤的數據、物理斷線）來測試系統的魯棒性和故障安全機制。

### **結論**

Vector Informatik 在 EV/EVSE 領域的解決方案展現了其在汽車電子通訊領域的深厚技術實力。從底層的物理訊號 (如 CP 訊號、車用乙太網路 SPE) 到高層的應用協議 (如 ISO 15118-20, OCPP)，Vector 都提供了強大的硬體介面、專業的軟體工具和豐富的協議堆疊支援。

然而，電動車充電通訊的技術門檻正隨著標準的演進和功能的複雜化而日益增高。開發者和測試工程師不僅需要精通各項協議的規範，更需掌握跨協議、跨硬體的整合與除錯技能。特別是 Plug & Charge 的複雜安全性實作、MCS 的兆瓦級功率傳輸挑戰、V2G 的多方協調以及車用乙太網路的實時性與訊號完整性，都要求極高的專業知識和先進的測試手段。

Vector Informatik 的產品，如 CANoe.SmartCharging 套件、VN/VT/VH 系列硬體，正是為了幫助產業應對這些挑戰而設計。它們提供了一個從早期開發、功能測試到系統級驗證的全面平台，使得工程師能夠更高效、更安全地開發和驗證新一代的電動車充電技術。隨著全球電動化轉型的加速，對這些通訊協議的深入理解和高效測試將成為產業成功的關鍵。
