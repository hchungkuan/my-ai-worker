## Vector Informatik EV/EVSE 通訊協定支援深度技術研究報告

### 前言

電動車 (EV) 與電動車充電設備 (EVSE) 的通訊協定是實現智慧充電、車輛到電網 (V2G)、即插即充 (Plug & Charge) 等先進功能的基石。Vector Informatik 作為汽車電子開發與測試工具的領導者，其產品和解決方案在 EV/EVSE 通訊協定棧的各個層次提供了全面的支援，從底層物理層到高層應用層，涵蓋了設計、模擬、分析、測試與驗證的整個生命週期。本報告將深入探討 Vector Informatik 在此領域所支援的通訊協定、對應硬體工具，以及開發者在實作與測試時可能面臨的深度技術門檻。

### **ISO 15118 系列 (包含 -2, -3, -4, -5, -20)**

**技術定義：**
ISO 15118 是一個國際標準系列，定義了電動車輛與充電設備之間的高階數位通訊協定，旨在實現智慧充電、雙向能量傳輸 (V2G/V2X)、即插即充 (Plug & Charge) 等功能。
*   **ISO 15118-2:** 規範了 AC 和 DC 充電的通用要求，包含負載調度、身份驗證等基本功能。
*   **ISO 15118-3:** 專注於物理層和資料連結層的電氣介面，特別是基於 HomePlug Green PHY (HPGP) 的電力線通訊 (PLC)。
*   **ISO 15118-4/-5:** 涉及一致性測試和協定實作指南。
*   **ISO 15118-20 (Unified Standard):** 是該系列的最新版本，旨在整合並取代早期標準。它支援 AC、DC、無線充電、兆瓦級充電系統 (MCS) 以及更先進的 V2G 服務，包括多重能量傳輸服務 (如交直流同時充電、多種功率模式)、強化安全性、更靈活的負載調度與協商機制。

**OSI 參考模型階層：**
應用層 (Layer 7)；底層網路層使用 TCP/IP (Layer 3/4)，資料連結層使用 Ethernet (Layer 2)，物理層使用電力線通訊 (PLC) 或直接 Ethernet (Layer 1)。

**Vector 硬體/工具對應：**

| Vector 硬體/工具 | 通訊介面 | 主要功能/備註 |
| :--------------- | :------- | :------------ |
| **VH6501**       | HomePlug Green PHY (PLC), Ethernet | 專為 ISO 15118 測試設計，提供 EV 或 EVSE 側的 HPGP 介面，可用於監聽、模擬 PLC 通訊。是測試 ISO 15118-3 物理層和資料連結層的關鍵。 |
| **VN5640 / VN5610A / VN5430 / VN5240** | Ethernet (100BASE-T, 1000BASE-T) | 若EV與EVSE之間使用直連乙太網（例如在測試實驗室或特定應用中），或用於監控協定棧的乙太網層。 |
| **VT System**    | Ethernet, CAN, Analog/Digital I/O | 在硬體在環 (HIL) 測試中，VT System 可整合 VH6501 及其他 VN 介面卡，模擬 EV 或 EVSE 的行為，測試其通訊模組。 |
| **CANoe** (with Option CarCharging) | (N/A) | 核心軟體，支援 ISO 15118-2/-20 協定的模擬、分析、診斷、測試自動化。提供專用的充電通訊協定解碼器、狀態機監控、測試庫。 |
| **CANalyzer** (with Option CarCharging) | (N/A) | 用於監聽和分析 ISO 15118 通訊流量。 |
| **vTESTstudio**   | (N/A) | 測試設計與自動化工具，可為 ISO 15118 協定創建複雜的測試序列。 |

**深度技術門檻分析：**

1.  **協定複雜性與狀態機管理：** ISO 15118-20 引入了多種充電服務 (AC/DC/WPT/MCS)、V2G 能量傳輸服務、進階負載調度等，其狀態機轉換複雜且嚴格。開發者需精確理解協定流程，處理各種邊緣情況和異常流程，確保通訊的穩定性與可靠性。
2.  **安全性與憑證管理 (TLS/PKI)：** 即插即充和 V2G 功能依賴於強大的安全性，涉及 TLS 加密、數位憑證的發放、驗證與撤銷。這要求開發者具備 PKI 基礎知識，並能處理憑證生命週期管理，實作安全韌體更新機制。
3.  **電力線通訊 (PLC) 的挑戰：** HPGP PLC 容易受到電網雜訊、阻抗不匹配、信號衰減的影響，導致通訊中斷或錯誤。測試環境需要精確控制雜訊源，並確保 VH6501 與 DUT 之間有良好的阻抗匹配。在實際部署中，電磁相容性 (EMC) 是一個巨大挑戰。
4.  **標準演進與互通性：** ISO 15118 標準仍在不斷演進 (從 -2 到 -20)，不同版本間的相容性與不同製造商實作的互通性 (Interoperability) 測試是關鍵挑戰。確保符合性並與市場上多樣化的 EV/EVSE 產品協同工作需要大量的測試驗證。
5.  **高功率充電 (MCS) 的整合：** 針對 MCS 的 ISO 15118-20 實作，除了通訊協定本身，還需考慮與大功率電力電子系統的緊密整合，如精確的溫度監測、電壓電流控制的協調，以及故障安全機制。

### **DIN 70121**

**技術定義：**
DIN 70121 是德國電動車輛與直流充電站之間的數位通訊協定標準，是 ISO 15118-2 的前身和基礎。它主要涵蓋了 DC 充電的基礎通訊流程，包括負載協商、身份識別和充電控制。

**OSI 參考模型階層：**
應用層 (Layer 7)；底層網路層使用 TCP/IP (Layer 3/4)，資料連結層使用 Ethernet (Layer 2)，物理層使用電力線通訊 (PLC) (Layer 1)。

**Vector 硬體/工具對應：**

| Vector 硬體/工具 | 通訊介面 | 主要功能/備註 |
| :--------------- | :------- | :------------ |
| **VH6501**       | HomePlug Green PHY (PLC) | 與 ISO 15118 類似，用於 DIN 70121 的 PLC 物理層通訊監聽與模擬。 |
| **CANoe** (with Option CarCharging) | (N/A) | 支援 DIN 70121 協定的模擬、分析、診斷與測試。 |
| **CANalyzer** (with Option CarCharging) | (N/A) | 用於監聽和分析 DIN 70121 通訊流量。 |

**深度技術門檻分析：**

1.  **與 ISO 15118 的相容性與轉換：** 由於 DIN 70121 是 ISO 15118-2 的前身，許多現有設備可能需要同時支援兩種協定或從 DIN 70121 升級到 ISO 15118。開發者需要處理兩種協定在功能和流程上的差異。
2.  **電力線通訊的挑戰：** 由於依賴 HPGP PLC，DIN 70121 同樣面臨與 ISO 15118 相同的 PLC 穩定性、雜訊抗干擾和電磁相容性問題。
3.  **功能限制與未來擴展性：** DIN 70121 不支援如即插即充、V2G 等進階功能，對於需要實現這些功能的系統，必須升級到 ISO 15118，這涉及通訊模組和軟體邏輯的重大修改。

### **CHAdeMO**

**技術定義：**
CHAdeMO (Charge de Move) 是一種由日本開發的直流快速充電協定，廣泛應用於全球部分地區。它主要利用 CAN 總線進行 EV 與 EVSE 之間的通訊，控制充電流程和狀態交換。

**OSI 參考模型階層：**
應用層 (Layer 7)；底層網路層使用 CAN (Controller Area Network) (Layer 2) 和物理層 (Layer 1)。

**Vector 硬體/工具對應：**

| Vector 硬體/工具 | 通訊介面 | 主要功能/備註 |
| :--------------- | :------- | :------------ |
| **VN16xx 系列 (VN1630A, VN1640A)** | CAN | 基礎的 CAN 介面卡，用於監聽、模擬 CHAdeMO 協定的 CAN 報文。 |
| **VN8912A / VN8910A** | CAN, CAN FD, Ethernet | 可作為獨立的介面硬體或與 CANoe/CANalyzer 結合使用，提供多通道 CAN/CAN FD 支援。 |
| **VN7570 / VN7600** | CAN, CAN FD | 高性能 CAN/CAN FD 介面。 |
| **CANoe** (with Option CAN/CAN FD) | (N/A) | 提供 CHAdeMO 特定診斷報文的解碼，狀態機監控，並可模擬 EV 或 EVSE 行為。 |
| **CANalyzer** (with Option CAN/CAN FD) | (N/A) | 用於監聽、分析和診斷 CHAdeMO 的 CAN 通訊。 |
| **vTESTstudio**   | (N/A) | 創建自動化測試用例，對 CHAdeMO 協定進行功能、效能與一致性測試。 |

**深度技術門檻分析：**

1.  **CAN 總線的限制：** CHAdeMO 依賴於 CAN 總線，其頻寬和資料負載量相對於 Ethernet 較低。在處理大量狀態資訊或需要快速響應的場景下，可能需要精細的報文管理和優化。
2.  **報文定義與版本管理：** CHAdeMO 協定有不同的版本 (如 0.9, 1.x, 2.x)，不同版本之間的報文定義和功能可能存在差異。開發者需要確保設備支援正確的版本，並處理向下相容性問題。
3.  **複雜的控制邏輯：** 儘管基於 CAN，CHAdeMO 的充電流程仍涉及複雜的狀態機和錯誤處理機制，需要開發者對協定細節有深入理解。
4.  **與其他充電標準的互操作性：** 與 CCS (基於 ISO 15118) 等其他主流標準的互操作性是市場上的一大挑戰。

### **IEC 61851-1 (控制導頻 - Control Pilot)**

**技術定義：**
IEC 61851-1 是一個國際標準，定義了電動車 AC 充電設備和車輛之間的基本物理和電氣介面，特別是其核心的「控制導頻 (Control Pilot)」信號。該信號透過脈衝寬度調變 (PWM) 來傳達 EVSE 的最大電流能力以及 EV 的充電狀態請求。這是所有 AC 充電的基礎，也是 DC 充電啟動前的物理握手環節。

**OSI 參考模型階層：**
物理層 (Layer 1) 和資料連結層 (Layer 2) (用於基本信號)。

**Vector 硬體/工具對應：**

| Vector 硬體/工具 | 通訊介面 | 主要功能/備註 |
| :--------------- | :------- | :------------ |
| **VT System**    | Analog I/O, Digital I/O, Ethernet, CAN | VT System 及其模組（如 VT2000/VT6000 系列的 I/O 板卡）可以模擬或測量 IEC 61851-1 的控制導頻信號（PWM 電壓、占空比），並與高階通訊協定（如 ISO 15118）的邏輯控制相結合，實現 HIL 測試。 |
| **VN 系列**      | CAN, Ethernet | 雖然 IEC 61851-1 本身不是 CAN/Ethernet 協議，但 VN 介面卡用於測試 EV/EVSE 內部控制單元 (ECU) 監控或響應控制導頻信號時所使用的 CAN/Ethernet 通訊。 |
| **CANoe** (with Option CarCharging) | (N/A) | 在 CANoe 環境中，可通過腳本或專用面板，結合 VT System 實現對控制導頻信號的生成、監控和模擬，並與上層協定行為同步測試。 |
| **vTESTstudio**   | (N/A) | 設計基於控制導頻信號變化來觸發或驗證上層協定行為的自動化測試用例。 |

**深度技術門檻分析：**

1.  **類比信號精確測量與生成：** 控制導頻是一個類比 PWM 信號，需要高精度測量電壓電平、頻率和占空比。雜訊、阻抗變化和電纜長度都可能影響信號品質。
2.  **時序與狀態轉換：** 協定定義了精確的時序要求和狀態轉換（如 A, B, C, D, E, F 狀態）。開發者需要確保 EV 和 EVSE 的控制邏輯能嚴格遵守這些時序，避免競爭條件或誤判。
3.  **故障檢測與安全機制：** 對於控制導頻的異常狀態（如斷路、短路、信號不穩定），EV 和 EVSE 需要迅速檢測並進入安全狀態，這涉及到硬體保護和軟體容錯邏輯的設計。
4.  **與高階協定整合：** IEC 61851-1 是物理層基礎，之上才能運行 ISO 15118 等數位通訊。開發者需要處理這兩個層次之間的協調與握手，確保充電流程的順暢啟動和結束。

### **OCPP (Open Charge Point Protocol)**

**技術定義：**
OCPP 是一個應用層協定，定義了電動車充電站 (Charge Point) 與中央管理系統 (Central System / CSMS) 之間的通訊方式。它使得充電站可以被遠端監控、管理、授權用戶、處理交易、進行韌體更新等。OCPP 支援多個版本，如 OCPP 1.6J 和最新的 OCPP 2.0.1。

**OSI 參考模型階層：**
應用層 (Layer 7)；底層網路層使用標準的網際網路協定，通常是透過 WebSockets 或 HTTP (Layer 3/4) 在乙太網 (Layer 2) 上傳輸。

**Vector 硬體/工具對應：**

| Vector 硬體/工具 | 通訊介面 | 主要功能/備註 |
| :--------------- | :------- | :------------ |
| **VN5640 / VN5610A / VN5430 / VN5240** | Ethernet (100BASE-T, 1000BASE-T) | 用於監控 EVSE 與 CSMS 之間基於乙太網的 OCPP 通訊，分析網路流量。 |
| **CANoe** (with Option Ethernet) | (N/A) | 可用於監聽、分析 OCPP 消息，尤其是作為診斷或整合測試的一部分。雖然 Vector 沒有專門的 "OCPP Option"，但其 Ethernet 分析能力可用於此類協議。可透過客製化面板或腳本來解碼和模擬 OCPP 流量。 |
| **CANalyzer** (with Option Ethernet) | (N/A) | 用於監聽、分析 OCPP 乙太網流量。 |

**深度技術門檻分析：**

1.  **網路穩定性與延遲：** OCPP 通訊依賴於網際網路連接。網路不穩定、高延遲或斷線都可能導致交易失敗或充電站離線。開發者需考慮網路重連、消息重試和緩存機制。
2.  **安全通訊 (TLS)：** OCPP 2.0.1 強烈推薦使用 TLS 加密通訊以保護數據隱私和完整性。實作 TLS 需要處理憑證管理、密鑰交換和安全策略。
3.  **版本兼容性與特性差異：** OCPP 1.6J 和 2.0.1 在功能和消息結構上有顯著差異。開發者需要確保充電站和 CSMS 之間支援兼容的 OCPP 版本，並處理不同版本之間的特性映射。
4.  **後端系統整合：** OCPP 只是充電站與 CSMS 之間的通訊。它需要與計費系統、用戶管理、能源管理系統等後端基礎設施無縫整合。這涉及複雜的 API 設計和數據交換。
5.  **測試自動化與擴展性：** 在大規模部署中，對 OCPP 協定進行全面的自動化測試，包括功能、性能、壓力測試，以及錯誤注入，是確保系統穩定運行的關鍵。

### **車載網路協議 (CAN/CAN FD, Automotive Ethernet: 100BASE-T1, 1000BASE-T1, 10BASE-T1S)**

**技術定義：**
雖然這些協議本身並非直接用於 EV 與 EVSE 之間的通訊，但它們是 EV 內部通訊的骨幹，對於電池管理系統 (BMS)、車載充電器 (OBC)、DC/DC 轉換器以及其他與充電相關的 ECU 之間的數據交換至關重要。它們間接支援了 EV 的充電功能。
*   **CAN / CAN FD:** 廣泛使用的串列總線，用於控制、感測器數據交換。CAN FD (Flexible Data-Rate) 提供了更高的數據傳輸速率和更長的數據場。
*   **Automotive Ethernet (100BASE-T1, 1000BASE-T1, 10BASE-T1S):** 針對車載環境優化的乙太網標準，採用單對非遮蔽雙絞線，用於實現更高的頻寬、更少的線束重量和更複雜的網路架構，支援區塊化架構、OTA 更新、高階輔助駕駛等。10BASE-T1S 是最新標準，用於更低速但成本敏感的邊緣設備。

**OSI 參考模型階層：**
物理層 (Layer 1) 和資料連結層 (Layer 2)。

**Vector 硬體/工具對應：**

**深度技術門檻分析：**

1.  **網路拓撲與帶寬管理：** 現代車輛具有複雜的網絡拓撲，包含多個 CAN/CAN FD 總線和乙太網網絡。開發者需要優化網絡設計，確保足夠的帶寬和實時性，特別是對於與充電安全和控制相關的數據。
2.  **電磁相容性 (EMC/EMI)：** 車載環境對電磁雜訊非常敏感。單對乙太網 (T1) 需要嚴格的線束設計和 EMC 測試，以確保通訊的穩定性和可靠性。
3.  **時間同步與實時性：** 在複雜的車載網絡中，各個 ECU 之間的時間同步至關重要，尤其是在充電功率分配和安全關斷等應用中。IEEE 802.1AS (gPTP) 等協議的實作具有挑戰性。
4.  **診斷與韌體更新 (UDS/DoIP)：** 基於這些網絡協議實作診斷服務 (UDS over CAN / DoIP over Ethernet) 和遠程韌體更新 (OTA) 需要深厚的協議棧知識和安全設計。
5.  **網路安全：** 隨著車載網絡的複雜化，網路安全變得越來越重要。保護 ECU 之間的通訊免受未經授權的訪問或篡改是關鍵挑戰。

### 總結

Vector Informatik 在 EV/EVSE 通訊協定支援方面提供了一套全面且深入的工具鏈。從底層的物理介面 (如 VH6501 的 PLC、VN 系列的 CAN/Ethernet) 到高層的應用協定 (如 CANoe 的 ISO 15118 Option)，其產品線涵蓋了 EV/EVSE 系統開發、測試與驗證的各個環節。

然而，無論協定本身如何標準化，開發者仍需面對諸多技術門檻：

*   **標準的持續演進與複雜性：** 特別是 ISO 15118-20 引入了更多功能和服務，要求開發者不斷學習和適應。
*   **物理層的挑戰：** 電力線通訊 (PLC) 的電磁相容性與信號完整性問題始終是難點。
*   **安全性與憑證管理：** 隨著智慧充電功能的普及，網路安全和憑證生命週期管理成為不可或缺但極具挑戰的一環。
*   **互通性與一致性：** 確保產品在多樣化的市場中與不同製造商的設備協同工作，需要大量的符合性與互通性測試。
*   **系統整合：** 將通訊協定與實際的電力電子硬體、BMS 控制邏輯以及後端雲端系統無縫整合，是實現智慧充電解決方案的最終目標。

透過 Vector 的專業工具，開發者可以有效應對這些挑戰，加速 EV/EVSE 系統的開發進程，並確保產品的可靠性、安全性和符合性。
