# Vector Informatik 在 EV/EVSE 支援的通訊協定技術研究報告

## 概述

Vector Informatik 作為汽車電子領域領先的工具和軟體解決方案供應商，其產品廣泛支援電動車 (EV) 與電動車供電設備 (EVSE) 之間以及內部系統的各種通訊協定。這些協定涵蓋了從底層物理介面到應用層的複雜通訊，對於實現安全、高效、智能的充電過程至關重要。本報告將深入探討 Vector Informatik 所支援的主要充電通訊協定，包括其技術定義、對應的硬體/工具，以及開發者在實作和測試這些協定時可能遇到的深度技術門檻。

## 1. ISO 15118 系列 (包含 DIN 70121, MCS, V2G 相關標準)

### 技術定義
ISO 15118 是國際電動車充電通訊標準系列，定義了電動車與充電站之間用於交流 (AC) 和直流 (DC) 充電的數位通訊協定。它不僅涵蓋了基本的充電協商，還支援更進階的功能，例如：
*   **Plug & Charge (PnC)**: 通過車輛與充電站之間的自動化安全認證和授權，實現「即插即充」的便利體驗。
*   **Vehicle-to-Grid (V2G) / Vehicle-to-Everything (V2X)**: 允許電動車在需要時向電網供電，或與其他外部設備交換能量，是未來智能電網的關鍵。
*   **Megawatt Charging System (MCS)**: 針對重型電動車輛（如電動卡車和巴士）開發的超大功率直流充電標準，基於 ISO 15118-20。
*   **DIN 70121**: 德國標準，是 ISO 15118-2 直流通訊協定的基礎和前身。

ISO 15118 系列通常基於 TCP/IP 協定棧，使用乙太網作為物理層。為了在資源受限的嵌入式系統中高效傳輸 XML 格式的訊息，它採用了 EXI (Efficient XML Interchanges) 編碼方式。TLS (Transport Layer Security) 用於端到端安全通訊。

| **表達層/會話層 (L6/L5)** | EXI 編碼/解碼, TLS 會話建立 | CANoe.SmartCharging (內建 EXI 處理), MICROSAR (TLS 堆疊, 加密模組)                                   |
| **傳輸層 (L4)**      | TCP, TLS (Transport Layer Security) | CANoe.SmartCharging (TCP/TLS 流量分析/模擬), MICROSAR (TCP/IP 堆疊, TLS 堆疊)                         |

### 深度技術門檻分析

1.  **標準複雜度與演進性**: ISO 15118 是一個龐大且持續演進的標準系列。理解其各個部分的交互作用（例如，從基本充電到 PnC、V2G 的狀態機轉換、訊息序列）需要深入研究。特別是 ISO 15118-20 引入的 MCS 和 V2G 更是增加了通訊的複雜性。
2.  **安全性 (TLS/PKI)**: PnC 和 V2G 依賴於強大的安全機制，包括 TLS 加密和基於公開金鑰基礎設施 (PKI) 的憑證管理。這要求開發者具備密碼學、憑證簽發與驗證、金鑰管理和安全漏洞防護的知識。實作和測試 TLS 握手、憑證鏈驗證等環節極具挑戰。
3.  **互操作性 (Interoperability)**: 由於不同廠商對標準的解釋和實作可能存在細微差異，導致 EV 與 EVSE 之間難以實現完美互通。這需要大量的測試場景覆蓋，包括錯誤注入、邊緣案例測試、壓力測試等，以確保穩定可靠的通訊。
4.  **即時性與性能**: V2G 應用對通訊的即時性有較高要求，特別是在電網調度或緊急情況下。如何在維持安全性的同時，優化通訊堆疊的性能，減少延遲，是一個技術難點。
5.  **EXI 編碼/解碼**: EXI 雖然高效，但其二進位格式的編碼與解碼邏輯相對複雜，且要求嚴格遵守模式定義。這增加了開發和除錯的難度，需要專門的工具支援。
6.  **測試環境建置**: 搭建一個能模擬各種充電情境 (包括錯誤、惡意行為、網路不穩定) 的測試環境成本高昂且複雜，需要多種工具和硬體的協同工作。

## 2. IEC 61851 (AC 充電控制)

### 技術定義
IEC 61851 是一個國際標準系列，定義了電動車充電系統的電氣特性，尤其是在 AC 充電方面。其核心是**控制導頻 (Control Pilot, CP) 信號**。CP 線上通過 PWM (脈衝寬度調變) 信號來傳輸電動車與充電樁之間的基本狀態資訊和最大允許充電電流。這是一個較為基礎的、基於模擬信號的通訊方式。

### 硬體/工具對應表

| 協定層級 (OSI Layer) | 協定/介面 | Vector Informatik 支援工具/介面                                              |
| :------------------- | :---------- | :--------------------------------------------------------------------------- |
| **應用層 (L7)**      | N/A         | N/A (基於物理層信號交互)                                                     |
| **資料連結層 (L2)**  | N/A         | N/A                                                                          |
| **實體層 (L1)**      | 控制導頻 (CP) PWM 信號 | CANoe (配合 VT System 或 VN 系列的模擬/數位 I/O 模組，例如 VN1630A/31A 的 CAPL 腳本解析) |

### 深度技術門檻分析

1.  **模擬信號處理與雜訊敏感性**: CP 信號本質上是模擬的 PWM 信號，容易受到電氣雜訊和干擾的影響。精確地測量、生成和解釋 PWM 信號的佔空比和電壓電平，需要精密的模擬輸入/輸出硬體和信號處理演算法。
2.  **時序精確性**: PWM 信號的佔空比直接決定了充電電流限制，任何微小的時序偏差都可能導致充電異常或安全問題。開發者需要確保 PWM 信號生成和採樣的極高時序精確性。
3.  **錯誤檢測與恢復**: IEC 61851 本身沒有定義複雜的錯誤檢測碼。當 CP 信號出現異常時，需要依賴上層應用邏輯或硬體設計來檢測並安全地處理這些錯誤狀態（例如，充電中斷）。
4.  **符合性測試**: 確保充電樁或車載充電器嚴格符合 IEC 61851 的電壓、頻率和佔空比要求，以確保與各種 EV/EVSE 的互操作性。

## 3. CHAdeMO (日本直流充電)

### 技術定義
CHAdeMO (Charge de Move) 是一個由日本推動的直流快速充電標準。它使用 CAN (Controller Area Network) 總線作為電動車與充電站之間的通訊介面，傳輸充電參數、狀態資訊、錯誤訊息等。CHAdeMO 的通訊協定相對成熟穩定，廣泛應用於亞洲和部分歐美國家。

### 硬體/工具對應表

| 協定層級 (OSI Layer) | 協定/介面            | Vector Informatik 支援工具/介面                                     |
| :------------------- | :------------------- | :------------------------------------------------------------------ |
| **應用層 (L7)**      | CHAdeMO 特定訊息     | CANoe (CHADemo 支援包), vTESTstudio (測試腳本), MICROSAR (CAN 應用層) |
| **資料連結層 (L2)**  | CAN Data Link Layer  | CANoe, VN系列 (CAN 介面卡), MICROSAR (CAN Driver/Stack)             |
| **實體層 (L1)**      | CAN Physical Layer (ISO 11898-2) | VN系列 (CAN 介面卡)                                               |

### 深度技術門檻分析

1.  **CAN 總線管理**: 雖然 CAN 是廣泛使用的技術，但在 CHAdeMO 中需要精確管理訊息的優先順序、週期發送、以及錯誤處理。當總線負載較高或存在錯誤幀時，如何維持通訊穩定是挑戰。
2.  **狀態機同步**: CHAdeMO 協定定義了複雜的充電狀態機。EV 和 EVSE 兩端必須嚴格同步其狀態轉換，任何不一致都可能導致充電失敗。除錯狀態機問題尤其困難。
3.  **錯誤處理與故障注入**: 如何模擬各種充電錯誤（例如，電壓過高/過低、電流異常、通訊中斷）並驗證系統的魯棒性，是測試的重點和難點。
4.  **向後相容性**: CHAdeMO 協定在不同版本之間可能存在細微差異，確保新舊充電樁和車輛的良好相容性需要額外的測試工作。

## 4. GB/T 27930 (中國直流充電)

### 技術定義
GB/T 27930 是中國的直流充電通訊協定標準，與 CHAdeMO 類似，它也採用 CAN 總線作為通訊介面。該標準定義了電動車與充電樁之間用於直流充電的訊息格式和通訊流程，是中國市場電動車充電的強制性標準。

### 硬體/工具對應表

| 協定層級 (OSI Layer) | 協定/介面            | Vector Informatik 支援工具/介面                                     |
| :------------------- | :------------------- | :------------------------------------------------------------------ |
| **應用層 (L7)**      | GB/T 27930 特定訊息  | CANoe (GB/T 支援包), vTESTstudio (測試腳本), MICROSAR (CAN 應用層) |
| **資料連結層 (L2)**  | CAN Data Link Layer  | CANoe, VN系列 (CAN 介面卡), MICROSAR (CAN Driver/Stack)             |

### 深度技術門檻分析

1.  **訊息解析與處理**: GB/T 27930 規定了詳細的 CAN 訊息格式，包括特定的位元編碼、資料單位和訊息 ID。開發者需要準確解析這些訊息並依據協定規範進行處理。
2.  **充電流程控制**: 該標準對充電握手、充電參數協商、充電控制和結束流程有嚴格的規定。如何設計穩健的控制邏輯，處理各種充電階段的切換和異常情況，是關鍵。
3.  **時序要求**: GB/T 協定對某些訊息的發送間隔和回應時間有明確要求，這對嵌入式系統的即時性提出了挑戰，尤其是在多任務環境中。
4.  **區域性差異與更新**: 儘管是國家標準，但實作中可能仍會遇到不同廠商對標準理解的差異，或標準本身的更新。及時跟進標準的變化並確保相容性是持續的挑戰。

## 5. OCPP (Open Charge Point Protocol)

### 技術定義
OCPP (Open Charge Point Protocol) 是一個應用層協定，主要用於**充電樁 (EVSE) 與其中央管理系統 (Central System)** 之間的通訊。它不是 EV 與 EVSE 之間的直接通訊協定，而是用於充電樁的遠端管理、狀態監控、充電事務處理、認證授權、固件更新等。OCPP 支持兩種主要格式：SOAP (基於 XML) 和 JSON (基於 WebSockets)。

| **表達層/會話層 (L6/L5)** | HTTP/WebSocket          | CANoe (網路流量分析), 特定網路模擬工具                                       |

| **網路層 (L3)**      | IPv4/IPv6               | CANoe (IP 封包分析), MICROSAR (TCP/IP 堆疊, 對於 EVSE 內部)                  |

| **實體層 (L1)**      | Standard Ethernet (100BASE-TX, 1000BASE-T) | VN系列 (Ethernet 介面卡), VT System                                      |

### 深度技術門檻分析

1.  **網路穩定性與連線管理**: OCPP 依賴於網際網路連線，因此網路不穩定、延遲、斷線等問題會嚴重影響充電樁的運營。如何設計魯棒的連線管理和重連機制是挑戰。
2.  **訊息複雜度與異步性**: OCPP 訊息種類繁多，且其通訊是異步的。開發者需要理解複雜的請求/回應流程、狀態同步機制，並妥善處理訊息的隊列和超時。
3.  **安全性 (TLS)**: 與 ISO 15118 類似，OCPP 通訊也需透過 TLS 加密以保護敏感資料。憑證管理、安全傳輸、防範中間人攻擊等是重要議題。
4.  **後台系統整合**: OCPP 只是通訊協定，真正的挑戰在於如何將充電樁與後台的計費系統、用戶管理系統、能源管理系統等進行無縫整合。這要求對整個生態系統有深入理解。
5.  **版本迭代與擴展性**: OCPP 標準持續演進 (OCPP 1.6, OCPP 2.0.1 等)，並支持多種擴展功能。確保系統能夠適應未來的版本和功能擴展，同時保持向下相容性，是一個持續的工程挑戰。

## 6. Automotive Ethernet (通用於 EV/EVSE 內部及 ISO 15118)

### 技術定義
Automotive Ethernet 是一種專為汽車環境設計的乙太網技術，用於滿足車輛內高速、低延遲、成本效益高的通訊需求。它採用單對非屏蔽雙絞線 (Single Pair Ethernet, SPE)，顯著減少了線纜重量和成本。常見的標準包括：
*   **100BASE-T1**: 傳輸速率 100 Mbps。
*   **1000BASE-T1**: 傳輸速率 1 Gbps。
*   **10BASE-T1S**: 傳輸速率 10 Mbps，支持多點總線拓撲，成本效益高，適用於更簡單的應用和感測器網路。
在 EV/EVSE 領域，它不僅是 ISO 15118 的物理層基礎，也用於車輛內部的高頻寬通訊，如 ADAS 資料、車載娛樂系統，以及電池管理系統 (BMS) 與充電控制器之間的高速通訊。

### 硬體/工具對應表

| 協定層級 (OSI Layer) | 協定/介面               | Vector Informatik 支援工具/介面                                                                        |
| :------------------- | :---------------------- | :----------------------------------------------------------------------------------------------------- |
| **應用層 (L7)**      | SOME/IP, DoIP, raw IP (用於 ISO 15118) | CANoe.Ethernet (訊息解析), PREEvision (網路設計), MICROSAR (應用軟體)                             |
| **傳輸層 (L4)**      | TCP/UDP                 | CANoe.Ethernet (流量分析), MICROSAR (TCP/IP 堆疊)                                                   |
| **網路層 (L3)**      | IPv4/IPv6               | CANoe.Ethernet (封包分析), MICROSAR (TCP/IP 堆疊)                                                   |
| **資料連結層 (L2)**  | Ethernet MAC (IEEE 802.3), VLANs | CANoe.Ethernet (MAC 層分析/錯誤注入), VN系列 (Ethernet 介面卡), MICROSAR (Ethernet Driver/MAC Layer) |
| **實體層 (L1)**      | 10BASE-T1S, 100BASE-T1, 1000BASE-T1 | VN系列 (Ethernet 介面卡, 如 VN56XX系列), VT System (HIL 平台), CANoe.Ethernet (物理層錯誤注入與監控) |

### 深度技術門檻分析

1.  **電磁相容性 (EMC/EMI)**: 單對線的乙太網對電磁干擾更敏感，確保在惡劣汽車環境下的信號完整性和 EMC 合規性是一大挑戰。這需要精密的硬體設計、屏蔽技術和嚴格的測試。
2.  **時間敏感網路 (TSN)**: 隨著 ADAS 和自動駕駛的發展，汽車乙太網需要支持時間敏感網路，以保證關鍵資料的即時傳輸。這涉及到複雜的定時同步 (如 PTP, Precision Time Protocol) 和流量整形機制。
3.  **網路拓撲與交換機管理**: 現代車輛的乙太網路會形成複雜的星形或環形拓撲，涉及多個乙太網交換機。配置和管理這些交換機，包括 VLANs、QoS (Quality of Service) 設置，以確保資料優先級和隔離，需要專業知識。
4.  **診斷與除錯**: 高速乙太網的診斷比傳統的 CAN 總線更複雜。追蹤網路中的延遲、丟包、錯誤幀，並識別故障點，需要進階的網路分析工具和技術。
5.  **安全性**: 乙太網為車輛引入了更廣泛的網路介面，也帶來了新的網路安全威脅。保護乙太網路通訊免受入侵和惡意攻擊，需要實作 IPsec、MACsec 等安全協定。

## 7. CAN/CAN FD (車輛內部通訊，用於 CHAdeMO/GB/T)

### 技術定義
CAN (Controller Area Network) 和 CAN FD (CAN Flexible Data-rate) 是汽車行業最廣泛使用的串列通訊總線。它們提供了一種高效、可靠的方式，讓車輛內部各個電子控制單元 (ECU) 之間交換資訊。
*   **CAN**: 原始標準，最大資料速率通常為 1 Mbps，資料幀長度最多 8 位元組。
*   **CAN FD**: CAN 的增強版本，在不改變 CAN 總線架構的基礎上，提供了更高的資料速率（高達 8 Mbps 或更高）和更大的資料負載（最多 64 位元組），以滿足現代車輛對頻寬日益增長的需求。
在 EV/EVSE 領域，除了用於 CHAdeMO 和 GB/T 的充電通訊外，CAN/CAN FD 也用於車輛內部的電池管理系統 (BMS)、車載充電器 (OBC)、DC/DC 轉換器、電動馬達控制器等關鍵部件之間的資料交換。

### 深度技術門檻分析

1.  **網路管理與診斷**: 大型 CAN/CAN FD 網路可能包含數十甚至上百個 ECU。有效的網路管理、啟動/關閉流程、診斷服務 (UDS, Unified Diagnostic Services) 的實作和測試是複雜的。
2.  **時序分析與負載優化**: 確保所有訊息在截止時間內可靠傳輸，避免總線擁塞，需要對總線負載進行精確分析和優化。CAN FD 的高速部分與傳統 CAN 的低速部分協同工作，對時序設計帶來額外挑戰。
3.  **錯誤處理與容錯**: CAN 總線本身具有錯誤檢測和仲裁機制，但開發者需要設計上層軟體來響應和處理各種總線錯誤（如位元錯誤、格式錯誤、CRC 錯誤），確保系統的魯棒性。
4.  **安全問題**: 雖然 CAN 總線最初設計時未考慮網路安全，但隨著車輛互聯化程度加深，CAN 總線的安全問題日益突出。如何實作基於硬體或軟體的安全機制 (如 MAC、加密)，防止未經授權的訪問或訊息篡改，是重要的研究方向。
5.  **分散式系統除錯**: 在一個由多個 ECU 組成的分散式系統中，定位 CAN/CAN FD 通訊問題可能非常困難，需要專業的分析工具和系統化的除錯方法。
